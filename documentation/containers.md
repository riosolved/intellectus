# Containers - how code execution works

Nothing generated by the model, and nothing you write, ever executes
directly on the host. Every run happens in a throwaway container: the
file's folder is mounted read-only at `/work`, there is no network (except
image builds), and the container is removed when the run ends.

## Reading the code

`atk/runtime/` is four small modules, one concept each - study in order:

| Order | Module | The concept |
|-------|--------|-------------|
| 1 | `engine.py` | Everything reduces to shelling out one nerdctl command and reading (exit code, output) |
| 2 | `images.py` | Images are frozen recipes: registered Containerfiles + derived dependency images |
| 3 | `sandbox.py` | The cage rules: read-only `/work`, `--network none`, `--rm` |
| 4 | `runners/` | The public face, one module per language: language -> (image, command) |

`containerfiles/` beside them holds the recipes.

## The engine

**containerd is the runtime**, driven through
[nerdctl](https://github.com/containerd/nerdctl) - the containerd
project's docker-compatible CLI. The toolkit looks for `nerdctl` on PATH
and nothing else; without it, code is not executed (challenges save
UNVERIFIED, reviews go static).

The one escape hatch is explicit configuration - any CLI that accepts
the same `build` / `run` flags works:

```
ATK_CONTAINER_ENGINE=docker
```

**Getting containerd + nerdctl on Windows:** the practical route is
[Rancher Desktop](https://rancherdesktop.io) with the containerd runtime
selected - it ships `nerdctl` on PATH and runs containerd inside WSL2.
(Docker Desktop also uses containerd internally, but drives it through
the `docker` CLI.)

## Registered images

Every language registers a **Containerfile** in `atk/runtime/containerfiles/`,
named `<name>.Containerfile` and tagged `atk-<name>` when built:

```
atk/runtime/containerfiles/
|-- python.Containerfile    -> atk-python   (alpine + python3 + py3-pip)
`-- cpp.Containerfile       -> atk-cpp      (alpine + g++)
```

Rules for a Containerfile:

- **Exactly alpine, mandatory** - the first line is
  `FROM alpine:<version>`. Not `-alpine` variants of other images
  (`python:3.13-alpine` is not allowed), not debian/ubuntu/`-slim`.
- **The Containerfile sets up the environment explicitly** - install the
  interpreter, compiler, and package manager the language needs via
  `apk add --no-cache ...` (e.g. `python3 py3-pip` for Python), and
  nothing else. The image exists to run one file, not to be a dev
  environment.

## Per-exercise dependencies are baked, not reinstalled

Python student code may declare a `requirements.txt` beside `code.py`.
Dependencies are NOT installed on every run - the first run bakes them
into a derived image:

```
FROM atk-python
COPY requirements.txt /tmp/requirements.txt
RUN pip install --no-cache-dir --break-system-packages -r /tmp/requirements.txt
```

tagged `atk-python-deps-<sha256 of requirements.txt, first 12>`. Every
subsequent `atk run` reuses that image instantly; editing the
requirements.txt changes the hash and triggers exactly one rebuild. Since
dependencies are baked at build time, the run itself keeps
`--network none`.

Images are **built automatically on first use**: when a runner needs
`atk-<name>` and the engine doesn't have it, the toolkit builds it from
the registered Containerfile. Delete an image to force a rebuild after
editing its Containerfile:

```
nerdctl rmi atk-cpp     # or docker/podman rmi
```

## Registering a new language's runtime

Part of [adding a language](adding-a-language.md):

1. Write `atk/runtime/containerfiles/<name>.Containerfile` on the thinnest base
   that provides the toolchain.
2. Add `atk/runtime/runners/<name>.py` following `runners/cpp.py` as
   the pattern (mount read-only, no network, artifacts to the
   container's `/tmp`).
3. Point the language module's `run_solution` / `run_code` at it.
